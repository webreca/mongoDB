SECTION 1.1: WHAT ARE LOGICAL OPERATORS?
================================================================================

DEFINITION:
  Logical operators combine multiple conditions in MongoDB queries
  using Boolean logic (AND, OR, NOR, NOT).

PURPOSE:
  ✓ Combine multiple filter conditions
  ✓ Create complex query logic
  ✓ Boolean AND, OR, NOR operations
  ✓ Negate conditions
  ✓ Multi-condition filtering

BASIC SYNTAX:
  { condition1, condition2, ... }     // Implicit AND
  { $or: [ condition1, condition2 ] } // Explicit OR
  { $nor: [ condition1, condition2 ] } // NOR
  { field: { $not: { operator } } }   // NOT


AVAILABLE OPERATORS (4 total):
  $and - ALL conditions must be true (implicit with comma)
  $or  - AT LEAST ONE condition must be true
  $nor - NONE of the conditions can be true
  $not - Negate/invert a condition


================================================================================
SECTION 1.2: BEGINNER - $AND (IMPLICIT)
================================================================================

DEFINITION:
  AND combines conditions where ALL must be true.
  Most common in MongoDB.

IMPLICIT SYNTAX (Preferred):
  { field1: value1, field2: value2, field3: value3 }

EXPLICIT SYNTAX (Rarely used):
  { $and: [ { field1: value1 }, { field2: value2 } ] }

MEANING:
  Document matches ONLY if ALL conditions are true.
  Missing one condition = no match.

BEGINNER EXAMPLE #1: Two conditions
────────────────────────────────────────────────────────────────────────────

Query:
  db.students.find({ age: 20, major: 'Data Science' })

Translation:
  Find students who are AGED 20 AND study DATA SCIENCE

Result:
  0-1 student (very specific combination)

Logic:
  age: 20 ✓ AND major: 'Data Science' ✓ → MATCH


BEGINNER EXAMPLE #2: Three conditions
────────────────────────────────────────────────────────────────────────────

Query:
  db.students.find({
    isInternational: true,
    gpa: { $gte: 3.9 },
    'scholarshipDetails.awarded': true
  })

Translation:
  International students AND honor level (GPA >= 3.9) AND on scholarship

Result:
  3-5 students

Logic:
  International ✓ AND honorGPA ✓ AND hasScholarship ✓ → MATCH


BEGINNER EXAMPLE #3: With comparison operators
────────────────────────────────────────────────────────────────────────────

Query:
  db.students.find({
    age: { $gte: 20 },
    gpa: { $lte: 3.5 }
  })

Translation:
  Students who are 20+ years old AND GPA at or below 3.5

Result:
  ~25-30 students

Logic:
  age >= 20 ✓ AND gpa <= 3.5 ✓ → MATCH


WHY IMPLICIT AND IS DEFAULT:
  ✓ Most queries need all conditions true
  ✓ Cleaner syntax
  ✓ More efficient
  ✓ No need for explicit $and


================================================================================
SECTION 1.3: BEGINNER - $OR (AT LEAST ONE)
================================================================================

DEFINITION:
  OR combines conditions where AT LEAST ONE must be true.

SYNTAX:
  { $or: [ condition1, condition2, ... ] }

MEANING:
  Document matches if ANY condition is true.
  One true = match, all false = no match.

BEGINNER EXAMPLE #1: Two values
────────────────────────────────────────────────────────────────────────────

Query:
  db.students.find({ $or: [ { age: 20 }, { age: 25 } ] })

Translation:
  Find students aged 20 OR aged 25

Result:
  4 students (2 of age 20, 2 of age 25)

Logic:
  age: 20 ✓ OR age: 25 ✓ → MATCH


BEGINNER EXAMPLE #2: Multiple majors
────────────────────────────────────────────────────────────────────────────

Query:
  db.students.find({
    $or: [
      { major: 'Data Science' },
      { major: 'Engineering' }
    ]
  })

Translation:
  Find students studying Data Science OR Engineering

Result:
  2 students (STEM fields)

Logic:
  DS major ✓ OR Engineering major ✓ → MATCH


BEGINNER EXAMPLE #3: Different field types
────────────────────────────────────────────────────────────────────────────

Query:
  db.students.find({
    $or: [
      { isInternational: true },
      { gpa: { $gte: 3.9 } }
    ]
  })

Translation:
  Find international students OR honor students

Result:
  35-40 students (30 international + 17 honor - overlap)

Logic:
  isInternational ✓ OR honorGPA ✓ → MATCH


BETTER APPROACH FOR SIMPLE VALUES:
────────────────────────────────────────────────────────────────────────────

Instead of:
  { $or: [ { age: 20 }, { age: 25 }, { age: 26 } ] }

Better use $in:
  { age: { $in: [20, 25, 26] } }

Why:
  ✓ More efficient
  ✓ More readable
  ✓ Faster performance


WHEN TO USE $OR:
  ✓ Different fields: { $or: [ { field1: v1 }, { field2: v2 } ] }
  ✓ Complex conditions: { $or: [ { f1: {...} }, { f2: {...} } ] }
  ✗ Simple values: Use $in instead

SECTION 2.1: INTERMEDIATE - $NOR (NONE TRUE)
================================================================================

DEFINITION:
  NOR combines conditions where NONE can be true.

SYNTAX:
  { $nor: [ condition1, condition2, ... ] }

MEANING:
  Document matches only if ALL conditions are false.
  One true = no match, all false = match.

OPPOSITE OF:
  $or (which requires at least one true)

INTERMEDIATE EXAMPLE #1: Exclude two conditions
────────────────────────────────────────────────────────────────────────────

Query:
  db.students.find({
    $nor: [
      { isInternational: true },
      { 'scholarshipDetails.awarded': true }
    ]
  })

Translation:
  Find students who are NOT international AND NOT on scholarship

Result:
  0-5 students (very specific: domestic without aid)

Logic:
  NOT international ✓ AND NOT on scholarship ✓ → MATCH


INTERMEDIATE EXAMPLE #2: Exclude ranges (middle band)
────────────────────────────────────────────────────────────────────────────

Query:
  db.students.find({
    $nor: [
      { gpa: { $gte: 3.9 } },  // NOT honor
      { gpa: { $lt: 3.0 } }     // NOT struggling
    ]
  })

Translation:
  Find students in middle range (3.0-3.89)

Result:
  ~25 students

Logic:
  NOT honorGPA ✓ AND NOT struggling ✓ → MATCH (middle band)


INTERMEDIATE EXAMPLE #3: Multiple exclusions
────────────────────────────────────────────────────────────────────────────

Query:
  db.students.find({
    $nor: [
      { major: 'Data Science' },
      { major: 'Engineering' },
      { isInternational: true }
    ]
  })

Translation:
  Find students who are NOT in STEM AND NOT international

Result:
  ~55+ students (most students)

Logic:
  NOT DS ✓ AND NOT Engineering ✓ AND NOT international ✓ → MATCH


EQUIVALENT WAYS TO WRITE:
────────────────────────────────────────────────────────────────────────────

Using $nor:
  { $nor: [ { isInternational: true }, { status: 'suspended' } ] }

Using explicit AND with $ne:
  { isInternational: { $ne: true }, status: { $ne: 'suspended' } }

Both work, but second is often clearer.


WHEN TO USE $NOR:
  ✓ Multiple exclusions (none should be true)
  ✗ Single exclusion: Use $ne instead
  ✗ Exclude specific value: Use $nin instead


================================================================================
SECTION 2.2: INTERMEDIATE - $NOT (NEGATE)
================================================================================

DEFINITION:
  NOT negates/inverts a condition.

SYNTAX:
  { field: { $not: { operator: value } } }

MEANING:
  Document matches if the condition inside $not is false.

INTERMEDIATE EXAMPLE #1: Negate comparison
────────────────────────────────────────────────────────────────────────────

Query (using $not):
  db.students.find({ gpa: { $not: { $gte: 3.9 } } })

Equivalent query (simpler):
  db.students.find({ gpa: { $lt: 3.9 } })

Result:
  43 students (not honor level)

Note:
  $not is redundant here, $lt is simpler


INTERMEDIATE EXAMPLE #2: Negate regex
────────────────────────────────────────────────────────────────────────────

Query:
  db.students.find({
    'name.first': { $not: /^D/ }
  })

Translation:
  Find students whose first name does NOT start with D

Use Case:
  Pattern negation (useful when regex is complex)

Result:
  58-59 students


WHEN TO USE $NOT:
  ✓ Complex pattern negation (regex)
  ✗ Simple comparisons: Use $lt, $ne instead
  ✗ Rarely needed in most queries


================================================================================
SECTION 2.3: INTERMEDIATE - COMBINING AND + OR
================================================================================

COMBINING $AND AND $OR:

When AND and OR are both needed:

PATTERN 1: AND with embedded OR
────────────────────────────────────────────────────────────────────────────

Query:
  db.students.find({
    age: 20,
    $or: [
      { major: 'Data Science' },
      { major: 'Engineering' }
    ]
  })

Translation:
  Students aged 20 AND (DS OR Engineering major)

Logic:
  age: 20 ✓ AND (DS ✓ OR Engineering ✓) → MATCH

Result:
  0-1 student


PATTERN 2: OR with embedded AND
────────────────────────────────────────────────────────────────────────────

Query:
  db.students.find({
    $or: [
      { gpa: { $gte: 3.9 } },
      { $and: [
          { age: { $lt: 23 } },
          { gpa: { $gt: 3.7 } }
        ]
      }
    ]
  })

Translation:
  Honor students OR (young AND talented but not honor level)

Logic:
  honor ✓ OR (young ✓ AND talented ✓) → MATCH

Result:
  20-25 students


PATTERN 3: Complex multi-condition
────────────────────────────────────────────────────────────────────────────

Query:
  db.students.find({
    isInternational: true,
    $or: [
      { gpa: { $gte: 3.9 } },
      { age: { $lt: 22 } }
    ]
  })

Translation:
  International students AND (honor OR very young)

Logic:
  international ✓ AND (honor ✓ OR young ✓) → MATCH

Result:
  ~5-8 students


OPERATOR PRECEDENCE:
  Implicit AND (comma) has HIGHER precedence than $or

  So: { field1: v1, $or: [...] }
  Means: (field1 = v1) AND (...OR...)
  NOT: (field1 = v1) OR (... AND ...)

SECTION 3.2: ADVANCED - OPERATOR PRECEDENCE
================================================================================

CRUCIAL CONCEPT:
  MongoDB evaluates implicit AND (comma) BEFORE explicit $or

This means:

Query:
  { field1: value1, $or: [ { field2: v2 }, { field3: v3 } ] }

Evaluates as:
  (field1 = value1) AND (field2 = v2 OR field3 = v3)

NOT as:
  (field1 = value1 AND field2 = v2) OR field3 = v3

PRECEDENCE DEMONSTRATION:
────────────────────────────────────────────────────────────────────────────

Example: Filtering international students with high or low GPA

Query:
  db.students.find({
    isInternational: true,
    $or: [
      { gpa: { $gte: 3.9 } },
      { gpa: { $lt: 3.0 } }
    ]
  })

Evaluation:
  (isInternational = true) AND (gpa >= 3.9 OR gpa < 3.0)

Result:
  International students who are either honor or struggling
  NOT domestic honor students (isInternational filter applies first)


EXPLICIT VS IMPLICIT:
────────────────────────────────────────────────────────────────────────────

These are equivalent:

Implicit (cleaner):
  { field1: v1, field2: v2, $or: [...] }

Explicit (rare):
  { $and: [ { field1: v1 }, { field2: v2 }, { $or: [...] } ] }

Use implicit version (more readable).

SECTION 3.3: ADVANCED - PERFORMANCE OPTIMIZATION
================================================================================

PERFORMANCE CONSIDERATIONS:

Implicit AND:
  ✓ Most efficient
  ✓ MongoDB optimizes naturally
  ✓ Use this as default

Explicit AND:
  ~ Same performance as implicit
  ~ Less readable
  ✗ Avoid (prefer implicit)

$OR:
  ⚠ Requires checking all conditions
  ~ Can be slow on large collections
  ✓ Create indexes on all $or fields

$NOR and $NIN:
  ✗ Cannot use indexes effectively
  ✗ Full collection scan required
  ✗ Avoid on large collections

$NOT:
  ⚠ May require full scan
  ~ Usually have simpler alternative
  ✓ Use simpler operator if available


OPTIMIZATION TIPS:
────────────────────────────────────────────────────────────────────────────

1. Create indexes:
   db.students.createIndex({ gpa: 1, isInternational: 1 })

2. Put most restrictive conditions first:
   Good: { gpa: { $gte: 3.9 }, major: { $in: [...] } }
   (honor is more restrictive than major)

3. Use $in instead of $or for simple values:
   Bad:  { $or: [ { age: 20 }, { age: 25 } ] }
   Good: { age: { $in: [20, 25] } }

4. Combine AND conditions implicitly (not explicit $and):
   Bad:  { $and: [ { f1: v1 }, { f2: v2 } ] }
   Good: { f1: v1, f2: v2 }

5. Test performance:
   db.students.find({...}).explain("executionStats")

SECTION 4.1: BEST PRACTICES
================================================================================

PRACTICE #1: Prefer Implicit AND
  Use: { field1: v1, field2: v2 }
  Not: { $and: [ { field1: v1 }, { field2: v2 } ] }
  Why: Cleaner, more readable

PRACTICE #2: Use $IN Instead of $OR
  Use: { status: { $in: ['active', 'pending'] } }
  Not: { $or: [ { status: 'active' }, { status: 'pending' } ] }
  Why: More efficient, more readable

PRACTICE #3: Index Fields in $OR
  Index: db.students.createIndex({ field1: 1, field2: 1 })
  Why: $or needs to check multiple paths

PRACTICE #4: Understand Precedence
  Remember: AND (comma) before OR
  So: { f1: v1, $or: [...] } means (f1=v1) AND (...OR...)

PRACTICE #5: Test Query Performance
  Use: db.students.find({...}).explain("executionStats")
  Check: executionStages shows if using index

PRACTICE #6: Keep Queries Readable
  Use: Clear logic structure
  Add: Comments for complex logic

PRACTICE #7: Avoid $NOR on Large Collections
  Problem: Cannot use indexes effectively
  Solution: Rewrite with simpler operators if possible

SECTION 4.2: QUICK DECISION GUIDE
================================================================================

Need ALL conditions true?
  → Use implicit AND (comma-separated)

Need ANY condition true?
  → Use $or

Simple value matching (any of these)?
  → Use $in (not $or)

Need NONE of conditions true?
  → Use $nor

Need to negate condition?
  → Use $ne, $lt, $gt (simpler)
  → Or $not if absolutely necessary

Complex nested logic?
  → Combine AND and OR with proper precedence
  → Test with EXPLAIN


================================================================================
SECTION 4.3: COMMON MISTAKES TO AVOID
================================================================================

MISTAKE #1: Precedence Confusion
  Wrong: Assuming { f1: v1, $or: [...] } means (f1=v1 OR ...) instead of (f1=v1 AND ...)

MISTAKE #2: Using $OR for Simple Values
  Wrong: { $or: [ { age: 20 }, { age: 25 } ] }
  Right: { age: { $in: [20, 25] } }

MISTAKE #3: Not Indexing $OR Queries
  Wrong: Using $or without indexes on fields
  Right: Create index on fields in $or

MISTAKE #4: Using $NOT When Simpler Operator Exists
  Wrong: { gpa: { $not: { $gte: 3.9 } } }
  Right: { gpa: { $lt: 3.9 } }

MISTAKE #5: Using $NOR on Large Collections
  Wrong: $nor on 1M+ documents without optimization
  Right: Rewrite with $and and $ne if possible

MISTAKE #6: Complex Queries Without Testing
  Wrong: Deploying complex logic without EXPLAIN
  Right: Test performance before production
