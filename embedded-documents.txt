SECTION 1: WHAT ARE EMBEDDED DOCUMENTS?
================================================================================

DEFINITION:
Embedded documents are documents nested within other documents. Instead of storing
related data in separate collections and referencing them, you store the data
directly inside the parent document.

EXAMPLE - Traditional SQL (Normalized):
  Table: students { id, name, age }
  Table: courses { id, studentId, courseCode, grade }

EXAMPLE - MongoDB (Embedded):
  Collection: students {
    _id: ObjectId,
    name: { first: "Daksh", last: "Chauhan" },
    age: 20,
    courses: [
      { courseCode: "DS-401", credits: 4, grade: "A" },
      { courseCode: "STAT-301", credits: 3, grade: "A-" }
    ]
  }

KEY DIFFERENCE:
In MongoDB, all student data is in ONE document. In SQL, you'd need to JOIN
two tables. MongoDB's approach is more efficient for this type of relationship.

SECTION 2: MONGODB EMBEDDED DOCUMENT LIMITS
================================================================================

LIMIT #1: MAXIMUM DOCUMENT SIZE
────────────────────────────────────────────────────────────────────────────
Size Limit:              16 MB (16,777,216 bytes)
What It Includes:        Everything in document (field names, values, nested docs)
Why This Matters:        Can't store massive nested structures
Impact on Student Data:  NONE (60 students = 36 KB total, 0.0025% of limit)

HOW IT AFFECTS YOU:
- All 60 students together = 36 KB
- Single student document = ~600 bytes
- Remaining capacity = 16 MB - 0.036 MB = 15.964 MB FREE
- Safety margin = 99% of limit remaining

EXAMPLE THAT WOULD HIT LIMIT:
- Embedding 4,000 high-resolution photos per student
- Embedding entire video files per document
- Storing millions of log entries in nested arrays


LIMIT #2: NESTING DEPTH
────────────────────────────────────────────────────────────────────────────
Nesting Depth Limit:     100 levels deep
What It Means:           Documents can be nested up to 100 levels
Why This Matters:        Excessively deep nesting is impractical
Impact on Student Data:  NONE (current depth = 3 levels)

CURRENT STUDENT DATA NESTING:
  Level 1: Root document (student record)
  Level 2: name object, courses array, campusLocation object
  Level 3: Individual course properties, coordinates array, scholarship fields

  Remaining capacity = 97 levels for deeper nesting


LIMIT #3: ARRAY SIZE LIMIT
────────────────────────────────────────────────────────────────────────────
Hard Limit:              NO hard limit on array length
Practical Limit:         Limited by 16 MB document size
Current Data:            Each student has 2 courses (very small arrays)

EXAMPLE:
- Could store 10,000 course entries per student (if each was ~1.6 KB)
- Current data: 2 courses per student = plenty of room for growth
- If expanded to 100 courses per student = still only ~6 KB per document


LIMIT #4: FIELD NAME LENGTH
────────────────────────────────────────────────────────────────────────────
Hard Limit:              Unlimited
Practical Limit:         Field names count toward 16 MB total
Recommendation:          Keep field names under 100 characters

GOOD FIELD NAMES:
  name, gpa, skills, enrollmentDate, campusLocation

BAD FIELD NAMES (Too Long):
  studentEnrollmentDateAndTimeWhenTheStudentFirstBecameAnActiveParticipant
  (This would waste bytes in every document)

EXAMPLE #1: EMBEDDED NAME OBJECT
────────────────────────────────────────────────────────────────────────────

Why Embed?
  - Student always has exactly one first name and one last name
  - Name is always accessed together with student record
  - Creates logical grouping of related data
  - Supports queries on individual parts (firstName: "Daksh")

Structure:
  {
    _id: ObjectId("..."),
    name: {
      first: "Daksh",
      last: "Chauhan"
    },
    age: 20,
    major: "Data Science"
  }

Benefits:
  ✓ Single query returns complete student info with name
  ✓ Natural data representation (how we think about names)
  ✓ Can query by first name: db.students.find({ "name.first": "Daksh" })
  ✓ Can update entire name atomically
  ✓ No separate lookup needed

Size Impact: 40-50 bytes per document

SECTION 5: EMBEDDING VS REFERENCING - WHEN TO USE WHICH?
================================================================================

EMBEDDING IS BEST FOR:
────────────────────────────────────────────────────────────────────────────
1. One-to-Few Relationships
   Example: Student has 2 courses
   Why: Few enough to keep in single document

2. Data Always Accessed Together
   Example: Student name, major, age accessed with courses
   Why: Single query is more efficient than multiple queries

3. Atomic Updates Needed
   Example: Update student + all their courses together
   Why: Single document update is atomic

4. Data Doesn't Grow Unboundedly
   Example: Courses array (fixed at 2 items)
   Why: Prevents document from growing past 16MB limit

5. Data Isn't Shared Across Documents
   Example: Each course belongs to one student
   Why: No data duplication concerns


REFERENCING IS BEST FOR:
────────────────────────────────────────────────────────────────────────────
1. One-to-Many Relationships
   Example: Teacher has 100 students
   Why: Too many to embed efficiently

2. Many-to-Many Relationships
   Example: Students and courses (many students take same course)
   Why: Embedding would cause massive data duplication

3. Large Related Data
   Example: Student photos, videos, documents
   Why: Would exceed 16MB document limit

4. Frequently Updated Independently
   Example: Course description changes, shouldn't update 100 student docs
   Why: Update efficiency and consistency

5. Shared Across Multiple Documents
   Example: Course description used by many students
   Why: Avoid duplication

WHAT IS NESTING DEPTH?
How many levels deep your documents are nested.

CURRENT STUDENT DATA DEPTH:

Level 1: Root Document
  {
    _id: ...,
    studentId: ...,
    name: { ... },          ← Level 2 starts here
    courses: [ ... ]        ← Level 2 starts here
  }

Level 2: First Nesting Level
  name: {
    first: "Daksh",         ← Level 3

  }

Level 3: Second Nesting Level
  courses: [
    {
      courseCode: "DS-401", ← Level 3
      grade: "A"            ← Level 3
    }
  ]

SECTION 8: BEST PRACTICES FOR EMBEDDED DOCUMENTS
================================================================================

1. MONITOR DOCUMENT SIZE REGULARLY
   Command: db.students.stats()
   Shows: Average document size, total size, growth trends
   Frequency: Weekly for growing collections
   Action: Alert if approaching 12MB

2. USE SCHEMA VALIDATION
   Enforce: maxItems on arrays, maxLength on strings
   Prevent: Unbounded growth of nested arrays
   Example: Limit skills array to max 10 items

3. ARCHIVE OLD DATA
   Move: Historical data to separate archive collection
   When: After 1 year of data accumulation
   Benefit: Keep active documents small and fast

4. SEPARATE LARGE FILES
   Don't Embed: Photos, videos, PDFs, large documents
   Instead Use: File storage (S3, Azure Blob, etc.)
   Benefit: Optimized for file handling, CDN integration

5. LIMIT ARRAY GROWTH
   Monitor: Growing arrays (logs, activities, posts)
   Consider: Separate collection if exceeds 1,000 items
   Example: Move old activity logs to history collection

6. INDEX NESTED FIELDS
   Command: db.students.createIndex({ "courses.grade": 1 })
   Improve: Query performance on embedded data
   Example: Finding all 'A' grades across all students

7. USE PROJECTIONS
   Bad:  db.students.find({ major: "Data Science" })
   Good: db.students.find({ major: "Data Science" }, { name: 1, gpa: 1 })
   Benefit: Faster queries, reduced network traffic

8. DOCUMENT YOUR SCHEMA
   Create: Schema documentation like this file
   Include: Field meanings, sizes, relationships
   Benefit: Team understands data structure

SECTION 9: OPTIMIZATION TIPS & TRICKS
================================================================================

TIP #1: USE PROJECTIONS TO FETCH ONLY WHAT YOU NEED
────────────────────────────────────────────────────────────────────────────

Bad Query (Fetches Everything):
  db.students.find({ major: "Data Science" })

Better Query (Fetch Only Name and GPA):
  db.students.find(
    { major: "Data Science" },
    { name: 1, gpa: 1, major: 1 }
  )

Even Better (Fetch Nested Fields):
  db.students.find(
    { major: "Data Science" },
    { "name.first": 1, "name.last": 1, gpa: 1 }
  )

Benefits:
  ✓ Smaller documents transmitted over network
  ✓ Faster response times
  ✓ Less memory usage
  ✓ Better for large collections


TIP #2: STORE CALCULATED VALUES TO AVOID COMPUTATION
────────────────────────────────────────────────────────────────────────────

Instead of calculating every time:
  db.students.aggregate([
    { $group: { _id: "$major", avgGPA: { $avg: "$gpa" } } }
  ])

Store in document (if updated infrequently):
  {
    _id: ...,
    name: { first: "Daksh", last: "Chauhan" },
    gpa: 3.85,
    calculatedAverageGPA: 3.85  ← Pre-calculated
  }

Benefits:
  ✓ Query returns immediately
  ✓ No need for aggregation pipeline
  ✓ Trade off: Storage vs computation speed


TIP #3: CONSIDER SEPARATE COLLECTIONS FOR LARGE ARRAYS
────────────────────────────────────────────────────────────────────────────

Current (Embedded, Good for 2-100 items):
  student: {
    _id: ...,
    name: "Daksh Chauhan",
    courses: [ { ... }, { ... } ]  ← 2 items
  }

Alternative (Referenced, Better for 1000+ items):
  student: {
    _id: ...,
    name: "Daksh Chauhan"
  }

  courseTaken: {
    _id: ...,
    studentId: ...,
    courseCode: "DS-401",
    grade: "A"
  }

When to Switch:
  - Array exceeds 1,000 items
  - Array updated frequently independent of student
  - Array size grows unboundedly
  - Need to query array items separately


TIP #4: REGULAR MAINTENANCE & CLEANUP
────────────────────────────────────────────────────────────────────────────

Monthly:
  1. Check collection stats: db.students.stats()
  2. Review largest documents: db.students.find().sort({size: -1}).limit(10)

Quarterly:
  1. Archive old data
  2. Rebuild indexes if fragmented

Annually:
